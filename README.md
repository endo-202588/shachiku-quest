## サービス概要

**社畜クエスト**は、チーム内の業務量を可視化し、助け合いを促進する業務管理アプリです。

### 解決したい課題

- 誰がどのくらい忙しいのか分からない
- 助けたいけど、誰が助けを求めているか分からない
- 業務の偏りが見えにくい
- 管理職が社員の状況を把握しづらい

### 提供する価値

- その日の業務量を「ステータスコード」で簡単に共有
- 助けられる人と助けが必要な人をマッチング
- RPGゲームの世界観で楽しく協力を促進
- 管理職は社員の状況を一目で把握できる

---

## このサービスへの思い・作りたい理由

普段誰がどのくらいの業務負担があるのか、なかなか把握できないことをもどかしく思っていました。

そこで、"猫の手も借りたい"人と、"助けられます"という人を楽しくつなげて業務量の平均化を図り、助けた人も経験値を積むことでレベルアップしていく様子を楽しめるアプリを作りたいと考えました。

実用的な面では、その日のチームメイトの業務量が可視化され、お休みも含めて社員の状況が把握できる、管理職にとっては簡単に社員の負担度がつかめる、というメリットがあります。

## サービスの利用イメージ

### 1. 朝の業務開始時

各社員（勇者）は、出勤したらその日の業務量を登録します。

**ステータスコード（業務量）**

- 🟢 **平和** - 余裕があります（ヘルプ不要）
- 🟡 **疲れ気味（0.5h）** - ちょっと忙しい
- 🟠 **やばいかも（1h）** - かなり忙しい
- 🔴 **きてます（1.5h）** - 助けてください！
- ⚫ **昇天（2h〜）** - 限界です！
- ○ **お休み** - お休み

### 2. シャチークの酒場（ヘルプマッチング）

助けられる人は「使える魔法（お手伝い可能時間）」を登録します。

**使える魔法**

- ⚡ **ベハーフ（0.5h）** - ちょっとだけお手伝い
- ✨ **ベホワン（1h）** - しっかりお手伝い
- 💫 **ベホワンハフ（1.5h）** - がっつりお手伝い
- 🌟 **ベホロング（2h〜）** - 全力でお手伝い

### 3. ヘルプ依頼（シャチークの酒場）

「シャチークの酒場」は、助けが必要な人と助けられる人をマッチングする場所です。

- 現在ヘルプが必要なタスクの表示
- ヘルプができる勇者の名前・使える魔法の表示

#### ヘルプ依頼の流れ

1. **ユーザーA（助けが必要な人）**

   - タスク「資料作成」をヘルプ依頼に登録（所要時間：1時間）
   - 「シャチークの酒場」で自分のタスクに合った魔法（時間）の勇者を選び、ヘルプ依頼を出す

2. **ユーザーB（助けられる人）**

   - 「使える魔法（お手伝い可能時間）」を登録
   - 「シャチークの酒場」でヘルプ依頼一覧を確認
   - ヘルプを依頼されるのを待つもしくは自分の「使える魔法（お手伝い可能時間）」に合ったタスクを選択
   - 依頼を受けるか、自分から「手伝う」ボタンを押して受諾

3. **タスク完了**
   - ユーザーBがタスクを完了
   - ユーザーAが完了確認
   - ユーザーBが「徳（ポイント）」を獲得し、レベルアップに繋がる

#### 補足

- 助けられる人は、特定のタスクを自分からピンポイントで助けることも可能
- 完了すると「徳（ポイント）」を獲得し、レベルアップに繋がる

### 4. レベルアップ

一定の徳を得ると、レベルがアップします！

**レベル一覧**

- 🌱 **旅人** - 初心者（0〜99徳）
- 🌿 **先輩** - 経験者（100〜299徳）
- 🌳 **先生** - ベテラン（300〜599徳）
- 🏆 **達人** - エキスパート（600〜999徳）
- 🙏 **仏** - マスター（1000徳〜）

## サービスの差別化ポイント・推しポイント

業務管理と「助け合い」を組み合わせた、RPG風チーム協力アプリです。

業務管理アプリのようなものは多く存在していますが、このように"助けを求める、助ける"といったアプリはあまり見られません。
しかもそれをRPGゲームの代表作"ドラゴンクエスト"のパロディのような形でゲーム仕様にし、社員が楽しみながら協力することを促進するという発想は非常にユニークであり、実用的でありながらもエンタメ性のあるアプリになることが期待できると思います。

## アーキテクチャ設計

本アプリでは、RailsのMVC構造を基本としつつ、責務の分離を意識した設計を行っています。

### Service Object

業務ロジックをControllerやModelに肥大化させないため、Service Objectを導入しています。

例：

- `DailyResetService`
  - ヘルプ依頼を日次でリセットする処理を担当
  - cron / whenever により自動実行

これにより、Controllerはリクエスト処理に集中し、ビジネスロジックの再利用性を高めています。

---

### 状態管理（State管理）

ヘルプ依頼は以下の状態遷移で管理されています。

open → matched → completed
↓
cancelled

これにより、ヘルプ依頼の進行状況を明確に管理しています。

---

### 通知システム

ヘルプ依頼の重要なイベントで通知が生成されます。

- マッチング成立
- ヘルプ完了
- ヘルプ要請クローズ

通知モデルを独立させることで、将来的な拡張（メール通知など）にも対応しやすい設計にしています。

---

### チャット機能

ヘルプ依頼ごとにチャットを作成し、ユーザー間のコミュニケーションを可能にしています。

HelpRequest
↓
Conversation
↓
Message

という構造にすることで、ヘルプ依頼ごとに独立した会話を管理できます。

---

### 定期処理（cron）

日次で実行される処理は `whenever` を利用し、cronで管理しています。

例：

- ヘルプ依頼の自動リセット
- マッチング状態の整理

---

### 認証設計

認証には `Sorcery` を利用し、以下のログイン方法を提供しています。

- メールアドレス + パスワード
- Google OAuthログイン

Googleログインユーザーは初回ログイン時に仮ユーザーを作成し、プロフィール登録後にアプリを利用可能にしています。

## インフラ構成

本アプリはDockerを用いて開発環境を構築し、Renderを利用してデプロイしています。

### 構成概要

```
User
  ↓
Render (Rails)
  ↓
PostgreSQL
  ↓
Cloudinary
```

---

### 開発環境

開発環境はDockerを利用し、以下のコンテナで構成しています。

- web（Railsアプリケーション）
- db（PostgreSQL）
- cron（wheneverによる定期処理）

これにより、環境差異の少ない開発環境を構築しています。

---

### デプロイ

アプリケーションは Render にデプロイしています。

- Render Web Service
- PostgreSQL
- 自動デプロイ（GitHub連携）

GitHubへpushすると自動でデプロイが実行されます。

---

### 画像ストレージ

ユーザーアイコンなどの画像は Cloudinary を利用して管理しています。

Railsの Active Storage と連携し、クラウドストレージに保存しています。

---

### 定期処理

日次処理は `whenever` を利用してcronで実行しています。

主な処理：

- ヘルプ依頼のリセット
- マッチング状態の整理
